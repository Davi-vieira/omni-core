### 2. Tecnologia e Arquitetura

#### A) Por que Python 3.12 e o que é o "Core"?
* **Python 3.12:** É a versão estável mais rápida da linguagem até hoje. Ela trouxe melhorias significativas no gerenciamento de memória e processamento assíncrono (importante para o Bot não travar enquanto processa dados).
* **O Core:** Em engenharia, o "Core" é o código que contém a lógica de negócio pura (regras de cálculo, impostos, validação de estoque). Ele é independente da interface. Se amanhã você quiser trocar o programa de computador por um app de celular, o "Core" permanece o mesmo.

#### B) PyQt6 vs. CustomTkinter (O Programa "Real")
* **PyQt6:** É o padrão da indústria (usado em softwares como Adobe e Maya). **Pontos fortes:** Suporte a temas complexos, tabelas pesadas com milhares de itens e segurança. Ele gera janelas nativas do Windows.
* **CustomTkinter:** É mais simples e moderno visualmente, mas menos potente para sistemas de gestão complexos (ERPs).
* **O Executável:** Sim, ambos permitem criar um `.exe`. Mas para ser um "programa real", usamos o **PyInstaller** para compilar e o **Inno Setup** para criar aquele instalador clássico que o cliente clica em "Próximo > Próximo > Concluir".

#### C) Banco de Dados: PostgreSQL ou SQLite?
* **SQLite:** É um arquivo local. **Por que usar?** Para o "Modo Offline". Se a internet cair, o programa continua salvando vendas no arquivo local.
* **PostgreSQL:** É um banco de dados de servidor. **Por que usar?** Se o cliente tiver 3 caixas (computadores) na mesma loja, todos precisam ler o mesmo estoque. O Postgres sincroniza tudo.
* **A Solução:** O projeto deve ser **Híbrido**. Ele salva no SQLite local e, assim que detecta internet, sincroniza com o PostgreSQL na nuvem (Cloud Sync).

#### D) A Técnica RAG e a Escolha da IA
* **O que é RAG (Retrieval-Augmented Generation):** Em vez de "treinar" a IA (o que custa milhões), nós damos a ela um "Manual do Proprietário" em tempo real. Quando o cliente pergunta algo, o sistema busca no banco de dados, entrega a resposta bruta para a IA, e ela "traduz" para o cliente de forma educada.
* **Por que Gemini 1.5 Pro?** Não é por "venda", é por **Janela de Contexto**. Ele consegue ler manuais inteiros de 1 milhão de palavras em segundos.
* **Existem melhores?** O **GPT-4o** é excelente em lógica pura, mas o custo por mensagem é mais alto. O **Claude 3.5 Sonnet** é o atual rei da codificação e raciocínio humano. O sistema deve ser **Agnóstico**, ou seja, você pode trocar de IA apenas trocando uma chave API.

### 1. O Coração do Sistema: Diagrama de Entidade-Relacionamento (DER)

Não podemos colocar tudo em uma tabela só. Usaremos a **Normalização** para garantir que o sistema não tenha erros de cálculo.

#### Tabela: `Vendas` (O cabeçalho da transação)
Esta tabela registra o "quem", "quando" e "quanto".
* `id` (UUID/PK): Identificador único universal.
* `data_hora` (Timestamp): Momento exato da venda.
* `cliente_id` (FK): Relaciona com a tabela de clientes (opcional para vendas de balcão).
* `vendedor_tipo` (Enum): 'IA' ou 'HUMANO' (Para saber quem fechou a venda).
* `status` (Enum): 'PENDENTE', 'CONCLUIDA', 'CANCELADA', 'ESTORNADA'.
* `total_bruto` (Decimal 10,2): Soma dos preços de tabela.
* `total_desconto` (Decimal 10,2): Total subtraído (seja por IA ou manual).
* `total_liquido` (Decimal 10,2): O valor que realmente deve entrar no caixa.

#### Tabela: `Itens_Venda` (O que saiu do estoque)
* `id` (PK)
* `venda_id` (FK): Relaciona com a venda acima.
* `produto_id` (FK): Relaciona com o estoque universal.
* `quantidade` (Decimal 10,3): Suporta quilos (1.500kg) ou unidades (1.000).
* `preco_unitario_no_ato` (Decimal): Importante salvar o preço daquele momento, pois o preço do produto pode mudar amanhã.

#### Tabela: `Pagamentos_Venda` (Suporte a Múltiplos Tipos)
Aqui é onde o sistema resolve o problema de: "R$ 10 no PIX e R$ 5 no Dinheiro".
* `id` (PK)
* `venda_id` (FK)
* `metodo` (Enum): 'PIX', 'CARTAO_CREDITO', 'CARTAO_DEBITO', 'DINHEIRO', 'FIADO'.
* `valor_pago` (Decimal)
* `status_confirmacao` (Boolean): No caso do PIX, só vira `True` quando o Webhook do banco confirmar.

#### Tabela: `Auditoria_IA` (A "Caixa Preta")
* `id` (PK)
* `venda_id` (FK)
* `log_conversa` (Text): O log da negociação no WhatsApp.
* `justificativa_desconto` (Text): Por que a IA deu 5%? (Ex: "Cliente recorrente, margem de lucro permitia").
* `hash_seguranca` (String/SHA-256): Um código gerado com base nos dados da venda. Se alguém tentar alterar o valor no banco de dados manualmente, o Hash não vai bater, acusando fraude.

### 2. Lógica de Atendimento: O Bot Vendedor

Para que o bot não seja apenas um "tirador de pedidos", ele deve seguir o **Fluxo de Máquina de Estados (FSM)**:

1.  **Estado de Sondagem:** O cliente pergunta o preço. A IA busca no banco e responde.
2.  **Estado de Oferta:** Se o cliente não responder em 2 minutos, a IA pergunta: *"Precisa de mais alguma informação sobre o [Produto]?"*.
3.  **Estado de Negociação (A Inteligência):**
    * O cliente pede desconto.
    * A IA executa uma Query interna: `SELECT margem_lucro FROM produtos WHERE id = X`.
    * Se a margem for > 20%, a IA tem autorização para dar até 5% de desconto para fechar a venda na hora.
4.  **Estado de Fechamento:** A IA gera o pedido no banco como 'PENDENTE' e envia o PIX. O estoque fica **Reservado** por 15 minutos. Se não houver pagamento, o estoque volta a ficar disponível.

### 3. Engenharia de Instalação e Segurança Sênior

#### Como o software "domina" o sistema?
O programa terá um **Serviço de Monitoramento de Arquivos**.
* O cliente define uma "Pasta de Notas".
* Sempre que ele salvar um PDF ou XML de compra lá, o programa detecta o novo arquivo, usa **OCR (Reconhecimento de Caracteres)** ou leitura de XML, e alimenta o estoque sem que o dono precise digitar nada.

#### Segurança e LGPD
* **Criptografia em Repouso:** O arquivo SQLite local será criptografado com **SQLCipher**. Mesmo que roubem o PC, ninguém lê os dados sem a chave do programa.
* **Identidade Soberana:** A IA não envia dados dos clientes para "aprender". O processamento usa o conceito de **Context Injection**, onde os dados são enviados apenas para processar a resposta atual e deletados da memória da IA imediatamente depois.

### 1. O "Cérebro" de Vendas (Cross-Selling Autônomo)

A IA não sugere produtos aleatoriamente. Ela usa **Regras de Associação** combinadas com o contexto do **RAG**.

* **Lógica de Afinidade:** O sistema analisa o histórico: *"70% das pessoas que compram Carne também levam Sal Grosso"*. Isso é o algoritmo *Apriori*.
* **A Execução da IA:** Se o cliente pede Carne pelo WhatsApp, a IA não apenas confirma. Ela consulta a "Tabela de Afinidades" e diz: *"Davi, separei aqui a Picanha. Temos um Sal de Parrilla que acabou de chegar e combina muito. Quer que eu adicione ao pedido por apenas R$ 15?"*
* **Upselling Técnico:** Se o estoque de um item de marca A está alto e o de marca B (mais caro) está baixo, a IA prioriza oferecer o que tem mais margem ou o que precisa girar rápido.

### 2. Licenciamento e Proteção do Código (Chave de Ativação)

Como o Python é uma linguagem interpretada (o código fica "visível"), usaremos técnicas de **Ofuscação e Compilação**.

* **O Segredo do Código (PyArmor + Cython):** Transformamos seus scripts `.py` em arquivos binários complexos (`.pyd` ou `.so`). Isso torna o código ilegível para humanos. Se alguém tentar abrir, verá apenas símbolos sem sentido.
* **Chave de Ativação (HWID Binding):** Ao instalar, o programa gera uma "Identidade Digital" baseada na Placa-mãe e no Processador daquele PC específico.
    * **Bloqueio de Pirataria:** Se o cliente copiar a pasta do programa para outro PC, o sistema detecta que o HWID não bate com a Chave de Ativação e se bloqueia automaticamente.
* **Servidor de Licenças:** Você terá um painel web central onde gera as chaves (ex: `AAAA-BBBB-CCCC`). Você define se aquela chave vale por 30 dias ou é vitalícia.

---

### 3. Segurança: Como Invadiriam e Como Defenderemos

Um sistema que lida com dinheiro e estoque atrai dois tipos de ataques:

| Tipo de Ataque | O Motivo | A Defesa (Implementação) |
| :--- | :--- | :--- |
| **Engenharia Reversa** | Roubar sua lógica de IA e vender como se fosse deles. | **Compilação Binária:** O código fonte não vai para o cliente, apenas o executável ofuscado. |

| **Intercepção de API** | Roubar sua chave do Gemini ou do WhatsApp. | **Environment Vault:** As chaves de API ficam criptografadas e só são descriptografadas na memória RAM durante o uso. |

### 4. Gestão de Múltiplos Computadores (Rede Local)

### 1. A API de Validação (Roda no seu Notebook)
Usaremos **FastAPI** (Python) por ser ultra-rápida e segura. Ela será o "porteiro" que diz quem entra e quem fica de fora.

```python
# No seu Notebook (Servidor de Licenças)
from fastapi import FastAPI, HTTPException, Header
from pydantic import BaseModel
import hashlib

app = FastAPI()

# Simulação de Banco de Dados (Em produção, use PostgreSQL)
DATABASE_LICENCAS = {
    "DV-2026-X89": {"hwid": "987A-BC54-XYZ", "status": "ativo", "cliente": "Loja do Davi"},
    "TESTE-999": {"hwid": None, "status": "pendente", "cliente": "Novo Cliente"}
}

class LicencaRequest(BaseModel):
    chave: str
    hwid: str

@app.post("/validar_licenca")
async def validar(req: LicencaRequest):
    licenca = DATABASE_LICENCAS.get(req.chave)
    
    if not licenca:
        raise HTTPException(status_code=404, detail="Chave não encontrada")
    
    # Se a chave for nova (hwid None), vincula ao primeiro hardware que usar
    if licenca["hwid"] is None:
        licenca["hwid"] = req.hwid
        return {"status": "vinculado", "mensagem": "Licença vinculada com sucesso!"}
    
    # Se o HWID enviado for diferente do registrado, bloqueia (Tentativa de Pirataria)
    if licenca["hwid"] != req.hwid:
        return {"status": "bloqueado", "mensagem": "Erro: Esta chave já pertence a outro computador."}
    
    if licenca["status"] != "ativo":
        return {"status": "expirado", "mensagem": "Licença inativa. Regularize o pagamento."}

    return {"status": "autorizado", "token_sessao": "HASH_DE_SEGURANCA_GERADO_NA_HORA"}
```

def validar_proposta_ia(produto_id, preco_ofertado, db_cursor):
    """
    Regra de Ouro: A IA nunca pode vender abaixo da margem mínima permitida.
    """
    db_cursor.execute("SELECT preco_custo, preco_venda, margem_minima_ia FROM produtos WHERE id = ?", (produto_id,))
    produto = db_cursor.fetchone()
    
    if not produto:
        return False, "Produto não encontrado."

    custo, preco_tabela, margem_ia = produto
    limite_minimo = preco_tabela * (1 - margem_ia)

    if preco_ofertado < limite_minimo:
        return False, f"Proposta de R${preco_ofertado} negada. Abaixo da margem de segurança."
    
    if preco_ofertado < custo:
        return False, "ALERTA: Tentativa de venda abaixo do custo detectada."

    return True, "Proposta aceita pelas regras de negócio."
```

        if estoque < qtd: return False, f"Estoque insuficiente. Disponível: {estoque}"

        # 3. REGRA DE OURO (Sua correção sutil e vital)
        limite_minimo = max(p_venda * (1 - m_ia), custo)
        
        if preco_ofertado < limite_minimo:
            return False, f"Preço de R${preco_ofertado:.2f} negado. Mínimo aceitável: R${limite_minimo:.2f}"

        # 4. TRANSAÇÃO ATÔMICA (Tudo ou Nada)
        conn.execute("BEGIN TRANSACTION")
        
        # Registrar Venda
        total = preco_ofertado * qtd
        cursor.execute("INSERT INTO vendas (total_liquido) VALUES (?)", (total,))
        venda_id = cursor.lastrowid
        
        # Baixar Estoque
        cursor.execute("UPDATE produtos SET estoque_atual = estoque_atual - ? WHERE id = ?", (qtd, produto_id))
        
        conn.commit()
        return True, f"Venda #{venda_id} de {nome} concluída! Total: R${total:.2f}"

    except Exception as e:
        conn.rollback()
        return False, f"Erro crítico na transação: {e}"

# --- SIMULAÇÃO DO FLUXO CLI ---
if __name__ == "__main__":
    conexao = inicializar_db()
    print("--- OMNI-CORE: MODO TERMINAL ---")
    
    # Simulando Cliente perguntando
    print("[BOT]: Olá! O Pão Francês está R$ 0.60. Quantos você deseja?")
    quantidade = 10
    
    # Simulando Negociação
    oferta = 0.55 # Cliente pede desconto
    print(f"[CLIENTE]: Faz 10 pães por R$ {oferta} cada?")
    
    sucesso, msg = realizar_venda(conexao, 1, quantidade, oferta)
    
    if sucesso:
        print(f"[SISTEMA]: {msg}")
    else:
        print(f"[SISTEMA]: {msg}")
```

### 2. Por que este código "Respira"?

* **Atomicidade:** O uso de `BEGIN` e `COMMIT` garante que você nunca registre uma venda sem tirar o pão do estoque. Se o computador desligar no meio, o banco volta ao estado anterior (Rollback).
* **Independência da IA:** Note que a função `realizar_venda` não sabe que o Gemini existe. Ela apenas recebe números. Isso permite que você teste toda a lógica do seu negócio sem gastar 1 centavo de API.
* **Fidelidade ao Dados:** O `limite_minimo` protege o lucro do seu cliente final. É isso que faz o software ser "Enterprise".
